Section 01

Architecture Overview

MediSphere uses a domain-driven microservices architecture where each service owns its data, exposes versioned REST/WebSocket APIs, and communicates asynchronously via Redis Streams for all non-transactional events. The entire stack is Python-first with FastAPI services, PostgreSQL as the system of record, and a React 18 PWA frontend.

The platform is divided into six bounded contexts: Identity & Access, Inventory & Stock, Sales & Billing, Prescription & Clinical, Distribution & Logistics, and Intelligence (AI/BI). Each context maps to one or more microservices with a dedicated PostgreSQL schema (separate schemas within one cluster for small-team operability, promotable to separate databases under load).

Design principle

Keep the engineering team's cognitive load low. Same stack (FastAPI + SQLAlchemy + Alembic + Celery) everywhere. One Kubernetes cluster. Separate schemas, not databases, until load demands it. Optimise for maintainability first, then performance.

Section 02

Microservice Descriptions

auth-svc — Identity & Access

# auth_svc/main.py — key routes
POST /auth/login # email+password → access_token (JWT, 15 min) + refresh_token (7 days)
POST /auth/refresh # rotate refresh token, issue new access token
POST /auth/totp/verify # second factor for admin/pharmacist roles
POST /auth/logout # revoke refresh token (stored in Redis blocklist)
GET /auth/me # decode + enrich token claims with branch_id, permissions
POST /auth/impersonate # super-admin only; bounded 1-hour token, fully audited

JWT access tokens carry: sub (user_id), role, branch_id, permissions[] (fine-grained), session_id (for audit trail). All sensitive operations validate the session_id against Redis to support instant revocation.

inventory-svc — Stock Management

The inventory service uses lot-based tracking with expiry dates. Each stock movement is an immutable ledger entry (similar to double-entry bookkeeping) — stock is never mutated in place, only new movements appended. Current stock levels are a materialised view (refreshed on write and by scheduled Celery task).

# Ledger-based inventory — no direct stock mutations
class StockMovement(Base):
__tablename__ = 'stock_movements'
id: UUID branch_id: UUID # FK → branches
product_id: UUID # FK → products
lot_id: UUID # FK → lots (barcode, expiry, supplier)
movement_type: Enum # RECEIVE|SALE|RETURN|TRANSFER_OUT|TRANSFER_IN|ADJUSTMENT|SHRINKAGE
qty: Decimal # signed: positive = in, negative = out
ref_id: UUID # FK to sale/PO/transfer that caused movement
performed_by: UUID created_at: DateTime # Indexes: (branch_id, product_id), (created_at DESC), (lot_id)

pos-svc — Point of Sale

All sales go through a two-phase commit pattern: HOLD → CONFIRM. A HOLD reserves stock optimistically; CONFIRM finalises payment and commits the ledger. This prevents double-sell at busy counters and supports offline operation — offline devices accumulate HOLDs locally, which resolve on reconnect with CRDT merge.

replenishment-svc — Procurement & Transfers

1

Trigger

Celery beat runs every 15 min comparing current stock vs reorder point (per product, per branch). AI forecast service adjusts reorder points weekly.

2

Draft order

System generates draft purchase order grouped by supplier. Branch manager reviews and approves via dashboard. Optional: auto-approve below configurable threshold.

3

Branch-to-branch transfer

If supplier lead time > urgency threshold, system checks sister branches for surplus. Transfer request sent; receiving branch confirms availability; distribution hub coordinates.

4

Fulfilment & GRN

Goods Receipt Note captured via barcode scan. Stock movements created for each received lot. Discrepancies flagged for manager review. Invoice matched automatically.

Key Design Decisions

  • All primary keys are UUID v7 (time-ordered) — sortable without performance penalty of random UUIDs on B-tree indexes.
  • All timestamps are TIMESTAMPTZ stored in UTC. Branch-local display handled in the application layer using pytz.
  • Sensitive fields (patient data, prescriber details, TOTP secrets) encrypted at rest using SQLAlchemy-Utils EncryptedType with AES-256.
  • Append-only audit_log table — never updated or deleted. Partitioned by month using PostgreSQL declarative partitioning.
  • PostgreSQL row-level security (RLS) policies enforce branch data isolation at the DB layer as a defence-in-depth measure.
  • All financial amounts stored as NUMERIC(12,4) — never FLOAT. Tax calculations performed server-side only.

API versioning strategy

All breaking changes increment the version path segment (/v1 → /v2). Non-breaking additions (new optional fields, new endpoints) do not increment. Deprecation notice period: minimum 60 days. Old versions kept alive for at least 1 year to support branch devices on slow update cycles.

Security Controls

Authentication

JWT (RS256) with 15-min access tokens. Mandatory TOTP for pharmacy_admin and pharmacist roles. Bcrypt password hashing (12 rounds). Account lockout after 5 failed attempts (15 min). Forced password rotation every 90 days.

Transport security

TLS 1.3 enforced end-to-end. HSTS with 1-year max-age. Certificate pinning for mobile PWA. Internal service mesh uses mTLS. No HTTP fallback permitted.

Data protection

AES-256 encryption for PHI/PII fields at rest. PostgreSQL RLS for row-level branch isolation. Secrets managed via Vault or cloud KMS — never in environment variables. S3/object storage encrypted at rest, pre-signed URLs only (1-hour expiry).

Audit trail

Every state-changing API call appends to audit_log with before/after JSONB snapshots. Audit log is append-only (no UPDATE/DELETE permissions). Tamper-evident via PostgreSQL trigger that blocks modifications. Auditor role has read-only cross-branch access.

Input validation

Pydantic v2 models on all API inputs. SQL injection prevented by SQLAlchemy ORM only (no raw f-strings). File uploads validated by magic-bytes, not extension. Prescription images virus-scanned before storage.

Regulated access controls

Scheduled drug dispensing requires dual-control: pharmacist authorisation + counter-staff confirmation. All controlled substance transactions flagged, timestamped, and reported to compliance module. Cannot be overridden offline.

Section 07

Offline Sync Strategy

Operational requirement

Several MediSphere branches have intermittent 2G/3G connectivity. The platform must continue handling OTC sales, stock lookups, and basic billing during disconnection. Reconnect sync must be safe, conflict-free for most operations, and transparent to staff.

Architecture

The offline strategy uses a three-layer approach: (1) a local SQLite database embedded in the PWA Service Worker via sql.js-httpvfs or wa-sqlite, (2) a client-side operation queue in IndexedDB, and (3) a server-side sync service that applies and reconciles operations on reconnect.

1

Local read model

At startup (or last successful sync), the PWA downloads a compressed SQLite snapshot of the branch's product catalogue, current stock levels, active pricing, and user session data. This snapshot is refreshed every 4 hours when online. Size target: <15 MB per branch.

2

Offline operation queue

Every mutation (sale, adjustment, return) is recorded as a serialised operation with: offline_id (UUID v7), type, payload, user_id, branch_id, device_time, sequence_no. Operations are written to IndexedDB atomically before UI confirmation.

3

Degraded mode guardrails

Controlled substances: dispensing blocked offline (regulatory requirement). Rx sales: allowed offline only for pre-verified prescriptions cached locally. Stock adjustments: buffered, but no new purchase orders can be raised offline. Inter-branch transfers: queued, dispatched on reconnect.

4

Sync-on-reconnect

On network restoration, the PWA posts the operation queue to POST /api/v1/sync/apply. The sync service applies operations in sequence-number order using last-write-wins for prices (server authoritative) and additive merge for stock movements (all movements applied, stock recalculated). Conflicts (e.g., selling an item another branch also sold offline) surface as manager alerts, not silent overwrites.

5

Idempotency

All sync operations use the offline_id as an idempotency key. The server stores processed offline_ids in a deduplication table (TTL 7 days). Re-submitting the same operation is safe. Network failures during sync are retried with exponential backoff (max 5 attempts, then manual resolution queue).

# sync_svc/apply.py — conflict resolution logic
async def apply_offline_operations(ops: list[OfflineOperation]) -> SyncResult:
conflicts = []
applied = []
for op in sorted(ops, key=lambda x: x.sequence_no):
if await dedup_store.exists(op.offline_id):
continue # already applied — idempotent skip
result = await apply_single(op)
if result.conflict:
conflicts.append(result) # surface to manager, don't block other ops
else:
applied.append(result)
await dedup_store.mark(op.offline_id)
return SyncResult(applied=applied, conflicts=conflicts)
Section 08

AI & Analytics Features

All AI features follow a human-in-the-loop design: the system recommends, the human decides. No AI action is irreversible without explicit user confirmation. All model outputs include an explanation and a confidence score. Staff can override any AI recommendation with a mandatory reason code (recorded to audit log).

Demand Forecasting

Prophet-based time-series model trained per SKU per branch, updated weekly via Celery scheduled task. Features include: historical sales (52 weeks), seasonality (festivals, flu season), promotions, regional weather proxy. Output: 30/60/90-day demand forecast with confidence intervals, fed into reorder point calculation.

Confidence threshold: 65% Human override: always Explanation: feature importance shown Audit: every suggestion logged

Substitution Recommendations

When a product is out of stock, the system suggests therapeutically equivalent alternatives ranked by: clinical equivalence score (static knowledge base), current stock availability, price proximity, and historic acceptance rate. Displayed at POS with pharmacist confirmation required for Rx substitutions.

Rx: pharmacist must confirm OTC: counter staff can accept No substitution for controlled drugs

Anomaly Detection

Isolation Forest model trained on transaction patterns (sale velocity, return rate, discount usage, quantity per transaction, time-of-day distribution). Flags suspicious patterns: unusually high returns, excessive discounts, off-hours bulk OTC sales (possible diversion), rapid stock reduction without corresponding sales. Alerts go to pharmacy_admin only — not visible to the implicated staff member.

Alert: admin only False positive rate monitored No automatic action Weekly model retrain

Conversational Query (NL → SQL)

Claude-powered natural language interface for operational data. User types a question; the system constructs a safe, read-only SQL query against the analytics read model, executes it, and returns the result with the SQL displayed for transparency. Queries are scoped by the user's branch_id by default. Prompt injection defended by schema-aware SQL generation with allow-listed tables and enforced WHERE clauses.

Read-only SQL only SQL always shown to user Branch-scoped by default No DDL/DML permitted Rate limited: 30 queries/hour

Section 10

Non-Functional Requirements

Performance

POS hold endpoint: p99 <200ms. Product search: p99 <150ms. Dashboard load: p95 <2s. All API responses <500ms p99 under normal load. Redis caches product catalogue and stock levels with 5-min TTL.

Scalability

Kubernetes HPA on all stateless services (CPU >70% → scale out). Peak evening traffic (6–9 PM): target 3× normal throughput. Seasonal surge (flu season, Diwali): pre-scale via scheduled HPA. DuckDB read replica scales independently from transactional DB.

Reliability

Target 99.5% uptime (allows ~44h/year maintenance). Postgres primary + 1 hot standby + 1 read replica. Automatic failover via Patroni (<30s RTO). Redis Sentinel for HA. Circuit breakers (Tenacity) on all inter-service calls.

Security

OWASP Top 10 baseline. Penetration test before go-live. Dependency scanning via Dependabot + pip-audit in CI. Secrets via Vault. No customer PII in application logs. WAF (ModSecurity) on Nginx edge.

Observability

Structured JSON logs (structlog) shipped to ELK. Prometheus metrics on all services (request rate, error rate, latency histograms). Distributed tracing via OpenTelemetry + Jaeger. Grafana dashboards per service + per KPI. Alertmanager PagerDuty integration.

Compliance

Patient data handling compliant with applicable health data regulations (DPDP Act for India). Prescription images stored encrypted, accessed only by authorised roles, purged per retention policy. Audit trail immutable, exportable for regulatory inspection.

Deployment Topology

Production runs on a managed Kubernetes cluster (EKS/GKE/AKS depending on provider). Each microservice is a separate Deployment with its own Horizontal Pod Autoscaler. The PostgreSQL cluster runs on dedicated VMs (not Kubernetes) for operational simplicity. All branch devices (tablets, POS terminals) are thin PWA clients — no server software on-premises, making branch IT overhead near-zero.

Small team consideration

With a small engineering team, avoid operational complexity: one Kubernetes cluster, one PostgreSQL cluster with schemas (not databases), one Redis cluster. Use managed services where available. Run Metabase as a sidecar, not a separate cluster. Consolidate logs in ELK rather than multiple tools.

CI/CD Pipeline

1

Push / PR

Lint (ruff), type check (mypy), unit tests (pytest). Dependency security scan (pip-audit). Docker build + push to registry on merge to main.

2

Staging deploy

Auto-deploy to staging on merge. Integration tests run against staging (pytest + httpx). DB migrations applied via Alembic. Smoke tests confirm all health endpoints.

3

Manual approval gate. Rolling update (zero downtime). Canary traffic split (10% → 50% → 100%) with automatic rollback on error rate spike. DB migration applied pre-deploy (backward-compatible only).

Section 12

Testing & Observability Plan

Testing Strategy

Unit tests (pytest)

Business logic: inventory calculations, FEFO lot selection, pricing, discount rules. Target 80% coverage on core service logic. All AI model decision boundaries unit-tested with fixture data.

Integration tests (pytest + httpx)

Full API round-trips against a test PostgreSQL instance. RBAC matrix fully tested — every role/endpoint combination. Offline sync scenarios tested using mock network layer.

E2E tests (Playwright)

Critical user journeys: POS sale, Rx dispensing, stock transfer, manager dashboard. Run nightly against staging. Mobile viewport tested explicitly (375px, 768px). Offline mode tested via Playwright's network interception.

Load tests (Locust)

Simulated peak evening load: 200 concurrent POS transactions across 42 branches. Ramp from 10 to 200 users over 10 minutes. Measure p99 latency. Run monthly and before seasonal periods.

Observability

  • All FastAPI services export Prometheus metrics via /metrics: request count, error rate, latency histogram, active DB connections.
  • Structured JSON logs with trace_id, span_id, user_id, branch_id, session_id on every log line — enables per-session audit trail debugging.
  • OpenTelemetry auto-instrumentation on all services; traces exported to Jaeger. Full end-to-end trace from browser → gateway → service → DB.
  • Custom Grafana dashboards: POS transaction rate, inventory adjustment spike detection, sync queue depth (offline branches), AI model inference latency.
  • SLO monitoring: 99.5% uptime, p99 <500ms error budget tracked weekly. Burn rate alerting via Alertmanager.
  • Offline branch health dashboard: shows last sync time per branch, queue depth, conflict count. Operations team can see at a glance which branches need attention.
Implementation phasing recommendation

Phase 1 (months 1–3): Auth, Inventory, POS, basic offline mode, 5 pilot branches. Phase 2 (months 4–6): Replenishment, Prescription, BI dashboards, all 42 branches. Phase 3 (months 7–9): AI features (demand forecast, anomaly detection, NL query), telehealth fulfilment. This lets the team validate the data model and sync strategy before adding AI complexity.
